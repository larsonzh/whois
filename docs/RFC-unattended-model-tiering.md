# RFC：A/B 无人值守模型分层与 GPT-5 mini 使用边界

## 1. 目的

本文用于固化 whois 仓库在 A/B 无人值守链路中的模型选型、思考档位和升级边界。
目标不是比较抽象模型能力，而是给出可直接执行的仓库内规则，降低长期运行成本，同时控制误判、误改和返工风险。

状态：可实施草案。

## 2. 范围

包含范围：
- A/B 无人值守运行中的会话内监控、工单处理、脚本修补与文档回填。
- `docs/RFC-whois-client-split.md` 与 `docs/RFC-address-space-preclassifier.md` 中的下次开工清单起草、任务定义编制与任务结束后回填。
- `testdata/` 下任务定义文件、active/smoke start-file 样例与相关说明文档的配套更新。

不包含范围：
- 对外发布文案、公告和 release body 的最终市场化润色。
- 与本仓库无关的通用模型对比。
- 替代现有脚本校验、静态体检和 gate 机制。

## 3. 模型分层

本文把会话中使用的模型分为两层：
- 低成本主模型：`GPT-5 mini`。
- 高能力升级模型：`GPT-5.4` 级模型。

设计原则：
- 默认优先使用低成本主模型，以控制长期 A/B 无人值守的运行成本。
- 一旦任务进入“多文件联动、语义改动、复杂判断、高返工成本”区域，必须升级到高能力模型。
- 模型切换依据任务风险和错误代价决定，而不是依据“当前修改量看起来大不大”。

## 4. GPT-5 mini 的默认思考档位

默认建议：`中档`。

档位定义：
- 低档：只适合执行型、模板化、低风险动作。
- 中档：适合作为无人值守日常主流程默认档位。
- 高档：适合设计、定稿、复杂修复和高风险判断。

默认规则：
- 无人值守主流程默认使用 `GPT-5 mini + 中档`。
- 仅当任务退化为“只读监控/回执/抄录结果”时，才允许降到低档。
- 仅当任务升级为“RFC 编制/任务设计/复杂修复/多文件联动决策”时，才切换到高档。

## 5. 适合交给 GPT-5 mini 的工作

### 5.1 可交给 GPT-5 mini 低档的工作

适用条件：
- 不改策略。
- 不改执行语义。
- 不做复杂根因判断。
- 改错后返工成本低。

典型工作：
- 轮询 `poll_agent_tickets.ps1` 并读取工单结果。
- 执行 `business_command`、`continue_watch_command`、`mark_processed_command`。
- 回填 `handled_at` 回执。
- 查看 queue、heartbeat、routine status、日志尾部。
- 核对文件是否存在、路径是否正确、样例是否在 active/smoke 目录中。
- 做只读状态汇总与简短结论回报。

### 5.2 可交给 GPT-5 mini 中档的工作

适用条件：
- 需要轻量判断，但仍可被现有模板、字段和脚本校验约束住。
- 允许小范围改动，且有清晰的落点与可执行验证。

典型工作：
- 日常无人值守默认监控与状态票处理。
- start-file 的局部字段调整。
- smoke/active 样例文件的补充、注释、链接化说明。
- 文档中的路径修正、说明补全、现成样例入口维护。
- 依据现有模板生成任务定义草稿或开工清单草稿。
- 任务结束后的常规回填与轻量整理。
- 小范围 PowerShell 脚本改动，并配合现有校验脚本验证。

### 5.3 可交给 GPT-5 mini 高档的工作

适用条件：
- 仍在仓库既有契约范围内。
- 任务主要是设计、编制、整理，而不是开放式重构。
- 有明确的模板、RFC 或既有样例可锚定。

典型工作：
- 在 `docs/RFC-whois-client-split.md` 中起草下一期 A/B 开工清单。
- 在 `docs/RFC-address-space-preclassifier.md` 中起草对应任务安排和实施说明。
- 编制 `testdata/autopilot_code_step_tasks_*.json` 一类任务定义文件。
- 生成或补齐 active/smoke 启动文件样例。
- 任务结束后回填两份 RFC 中的执行结论、结论摘要与后续待办。

质量预期：
- 在高档下，GPT-5 mini 可以产出高可用的文档与任务设计草稿。
- 但产出仍必须经过最小校验，不能把“能胜任”理解为“无需复核”。

## 6. 不能交给 GPT-5 mini 的工作

以下工作不应由 GPT-5 mini 独立承担，必须升级到 `GPT-5.4` 级模型或同等级高能力模型。

### 6.1 高错误代价的开放式代码修复

典型特征：
- 无明确落点，需要跨多个模块搜索真正控制逻辑。
- 改动可能破坏输出契约、重定向语义、CIDR 行为或 IPv4/IPv6 规则。
- 验证失败后需要迅速调整假设并重新定位控制路径。

示例：
- 影响 whois 查询链路、权威判定、转发策略的根因修复。
- 跨 `src/core/`、`src/cond/`、`include/` 的行为级修复。
- 需要同时保持 stdout/stderr 契约稳定的复杂改动。

### 6.2 缺少稳定模板约束的复杂方案设计

典型特征：
- 不是在既有模板上增量编制，而是重新发明流程、字段或语义。
- 需要在多种可行方案中做体系级取舍。

示例：
- 重新设计无人值守状态机或 ticket 生命周期。
- 新建一套与现有 `AI_CHAT_POLICY_*` 并列的策略体系。
- 大幅调整 A/B 运行制度、重启屏障或自动恢复边界。

### 6.3 需要强一致性的跨工件变更统筹

典型特征：
- 同时涉及脚本、start-file、RFC、使用文档、黄金样例和回填口径。
- 任一处遗漏都会导致后续无人值守运行或复盘混乱。

示例：
- 改动 `AI_CHAT_POLICY_WORK_MODE` 的含义并同步所有文档、样例与脚本。
- 修改 A/B 起草、生成、回填的字段契约，并要求全仓一致。

### 6.4 不确定性高的根因分析与升级决策

典型特征：
- 日志表征不一致。
- 可能是脚本问题、模型执行问题、环境问题或业务逻辑问题叠加。
- 错误决策会造成长时间空转、误重启或错误回填。

示例：
- 判断编译失败是否应自动修复还是必须人工接管。
- 判断 B 轮失败后是否应从 A 快照恢复、仅重启 B，还是回到 A-D1。
- 判断终端批量消失是 VS Code 层异常还是业务链路真实失败。

## 7. 必须升级到 GPT-5.4 级模型的触发条件

出现以下任一条件，必须从 `GPT-5 mini` 升级：
- 需要跨多个模块重新定位真实控制逻辑。
- 需要改动输出契约、权威判定、RIR 重定向、CIDR 语义或 IPv4/IPv6 规则。
- 需要设计全新字段、全新流程或新的自动恢复机制。
- 需要在多个 RFC、多个样例文件和多个脚本之间做强一致统筹。
- 当前验证结果与原假设冲突，且无法通过一次邻近读取快速消歧。
- 错误决策的代价高于单次模型成本，例如会导致长跑任务空转、错误重启或错误回填。

## 8. 文档与任务编制工作的推荐做法

对于“两个 RFC 的下次开工清单、A/B 任务定义文件、任务结束后回填”这类工作，推荐流程是：
- 起草阶段：`GPT-5 mini + 高档`。
- 交叉核对阶段：对照现有 active/smoke 样例、既有窗口和任务模板做字段一致性检查。
- 落库前验证：至少执行一轮现有脚本校验或文件诊断检查。

最低质量门槛：
- 任务定义不得残留 TODO。
- 文档引用的样例路径必须可打开。
- start-file 相关修改后应通过 `tools/test/check_unattended_start_field_sync.ps1`。
- 若涉及脚本改动，目标文件诊断应为无错误。

## 9. 仓库内推荐运行策略

推荐默认策略：
- 日常无人值守：`GPT-5 mini + 中档`。
- RFC 起草、任务定义编制、复杂文档整理：`GPT-5 mini + 高档`。
- 复杂修复、开放式设计、跨模块根因分析：升级到 `GPT-5.4` 级模型。

不推荐的策略：
- 用 `GPT-5 mini + 低档` 长时间承担完整无人值守主流程。
- 让 `GPT-5 mini` 独立决定高错误代价的重启、回滚和体系级语义变更。

## 10. 验收标准

若采用本文策略，应满足：
- 日常无人值守成本明显低于长期使用高能力模型常开。
- 文档、start-file、任务定义的返工率保持在可接受范围。
- 遇到高风险任务时，升级边界清晰，不因“节省一点模型成本”而放大返工成本。
- 回填后的 RFC 与实际样例、脚本和运行口径一致。

## 11. 建议结论

结论如下：
- `GPT-5 mini` 适合作为 whois 仓库 A/B 无人值守的低成本主模型。
- 默认思考档位应为中档；文档/任务编制工作建议切高档。
- 对于两个 RFC 的开工清单起草、任务定义文件编制和任务结束后的回填，`GPT-5 mini` 能胜任，但必须保留最小校验。
- 对于开放式复杂修复、体系级方案设计、强一致跨工件统筹和高错误代价升级决策，应升级到 `GPT-5.4` 级模型处理。
